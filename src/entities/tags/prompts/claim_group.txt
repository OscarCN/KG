
Tarea: mantener el catálogo de claims (alegaciones factuales) que son relevantes para una entidad (persona, organización, o entidad en general) dentro de un evento específico.

Un claim es una afirmación factual relevante para la entidad dentro del evento.
Un cluster es una agrupación canónica de claims que dicen LO MISMO; tiene un `id`, un `canonical` (frase representativa) y `n_members` (claims asignados hasta ahora). El catálogo es la lista de clusters del evento.
Para cada claim nueva, rutéala a un cluster (assign), abre uno nuevo (create), o descártala (drop). Adicionalmente puedes proponer mutaciones al catálogo: renombrar el canonical de un cluster, o fusionar dos clusters.

Objetivos del catálogo:
- agrupar claims que dicen LO MISMO bajo un solo cluster, aunque la redacción varíe
- canonical breve, neutro y reusable; describe la alegación, no un caso particular
- no fusionar clusters con alegaciones distintas aunque toquen el mismo tema
- descartar lo que no es alegación factual concreta

debes leer cada claim nueva, decidir su ruta, y opcionalmente proponer mutaciones cuando aplique

se muestran los clusters existentes y las claims nuevas a rutear

REGLAS

- Una decisión por cada claim del lote, identificada por su `idx`.
- Prioridad cuando dudes: assign > create > drop. Sólo crea cluster nuevo si ninguno existente cubre la alegación.
- `assign` requiere un `cluster_id` válido de los clusters existentes.
- `create` requiere `canonical` reusable: frase corta (≤ 12 palabras), neutra, que sirva para futuros claims similares.
- `drop` se usa para claims vagos, fuera de tema, opinión, sarcasmo, o duplicados de otra claim del mismo lote.
- `merge` sólo si dos clusters describen LA MISMA alegación; alegaciones parecidas con matices distintos NO se fusionan. Sé conservador.
- `rename` sólo si el canonical actual quedó impreciso, sesgado o demasiado largo.

idx: índice de la claim.
action: "assign" | "create" | "drop".
cluster_id: id válido cuando action="assign"; null en otros casos.
canonical: frase canónica cuando action="create"; null en otros casos.
reason: ≤ 15 palabras (español de México) que justifiquen la decisión.
kind: "rename" o "merge" en `mutations`.
new_canonical: reemplazo del canonical en una mutación rename.
src_id / dst_id: clusters origen y destino de un merge; src se vacía hacia dst.


Responde EXCLUSIVAMENTE con JSON:

{
  "decisions": [
    {"idx": 0, "action": "assign|create|drop", "cluster_id": "<id-o-null>", "canonical": "<o-null>", "reason": "..."}
  ],
  "mutations": [
    {"kind": "rename", "cluster_id": "<id>", "new_canonical": "..."},
    {"kind": "merge", "src_id": "<id>", "dst_id": "<id>"}
  ]
}

Ejemplo (claim 0 cae en un cluster existente; claim 1 abre cluster nuevo; claim 2 es opinión y se descarta; además se renombra un cluster con canonical impreciso):

{
  "decisions": [
    {"idx": 0, "action": "assign", "cluster_id": "c_demora_pagos", "canonical": null, "reason": "misma alegación de retraso"},
    {"idx": 1, "action": "create", "cluster_id": null, "canonical": "cierre de sucursales en el trimestre", "reason": "alegación nueva no cubierta"},
    {"idx": 2, "action": "drop", "cluster_id": null, "canonical": null, "reason": "opinión sin hecho verificable"}
  ],
  "mutations": [
    {"kind": "rename", "cluster_id": "c_demora_pagos", "new_canonical": "demora en pago de siniestros"}
  ]
}

Si no hay decisiones ni mutaciones, devuelve {"decisions": [], "mutations": []}.
